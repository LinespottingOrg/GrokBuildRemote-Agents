# Grok Build Remote — Relay Protocol v1

**Status:** LOCKED for Day-1 implementation  
**Brand:** Grok Build Remote  
**Transport:** HTTPS JSON via xAI Grok API (no phone↔PC sockets)

## Design goals

1. Phone and PC never open ports to each other
2. Every message is scoped to `device_id` + `session_id`
3. Idempotent injects (`command_id`) so offline queues are safe
4. Works if Grok only provides chat completions (envelope-in-JSON fallback)

## Identifiers

| ID | Format | Meaning |
|----|--------|---------|
| `device_id` | UUID v4 | One PC installation |
| `session_id` | slug `^[a-z0-9][a-z0-9-]{1,62}$` | Named terminal (e.g. `global-edition`) |
| `command_id` | UUID v4 | One inject attempt |
| `pairing_code` | 8-char Crockford base32 | One-time mobile↔agent link |

## Auth

- Agent reads Grok/xAI API key from `~/.grok/config.json` (or `%USERPROFILE%\.grok\config.json`)
- Mobile uses same X/Grok account login OR pairs with `pairing_code` + API key from user settings
- Day-1: API key in mobile secure storage is acceptable; OAuth polish later

## Envelope (all payloads)

```json
{
  "proto": "gbr/1",
  "type": "inject | output | register | list | heartbeat | pair",
  "device_id": "...",
  "session_id": "...",
  "command_id": "...",
  "ts": "2026-07-15T17:00:00Z",
  "payload": {}
}
```

## Message types

### `pair`
Mobile shows code; agent confirms.

```json
{ "type": "pair", "payload": { "pairing_code": "A1B2C3D4", "device_name": "DESKTOP-1" } }
```

### `register`
Agent advertises a session.

```json
{
  "type": "register",
  "session_id": "global-edition",
  "payload": {
    "cwd": "C:/Users/User/.aiprojects/gbr",
    "shell": "pwsh",
    "os": "windows",
    "title": "Windows Terminal",
    "git_remote": "LinespottingOrg/GrokBuildRemote-Agents"
  }
}
```

### `list`
Mobile requests sessions for `device_id`. Agent responds with `sessions[]`.

### `inject`
Mobile → agent: run/type into session.

```json
{
  "type": "inject",
  "session_id": "global-edition",
  "command_id": "…",
  "payload": {
    "mode": "text | nl",
    "text": "grok \"fix the tests\"",
    "nl_prompt": "fix the tests",
    "submit": true
  }
}
```

- `mode=text`: type exact string (+ Enter if `submit`)
- `mode=nl`: agent may pass to local Grok Build CLI as natural language (future); Day-1 treats as text unless CLI wrapper present

### `output`
Agent → mobile: terminal chunk.

```json
{
  "type": "output",
  "session_id": "global-edition",
  "command_id": "…",
  "payload": {
    "stream": "stdout | stderr | system",
    "chunk": "...",
    "eof": false
  }
}
```

### `heartbeat`
Agent every 30s: alive + session count.

## Transport modes

### Mode A — Preferred (tool calls)
If/when Grok supports durable tools:

- `gbr_register_session`
- `gbr_list_sessions`
- `gbr_inject_to_session`
- `gbr_post_output`

### Mode B — Day-1 mailbox (chat completions)
1. Pairing creates a shared **mailbox conversation** id stored on both sides
2. Mobile appends envelopes as JSON-only messages tagged `[GBR]`
3. Agent polls every 2–5s for new `[GBR]` messages addressed to it
4. Agent replies with `[GBR]` output envelopes

### Mode C — Direct xAI API both sides (same account)
Both use API key; agent hosts local **loopback control** only; remote path still Grok-mediated via Mode B until product tools exist.

## Session naming (agent)

Priority:
1. File `.grok-session` in cwd (first line = session_id)
2. Explicit rename command `gbr rename <id>` / `grok rename <id>`
3. Fallback: `{folder-name}` or `{folder}-{git-repo-name}` slugified

Scan interval: 5s.

## Security

- Never log full API keys
- Pairing codes expire 10 minutes
- Inject only into sessions owned by same `device_id`
- Reject path traversal / session_id with `/` or `..`
- Mobile offline queue persists encrypted at rest when OS supports Keychain/Keystore

## Versioning

Breaking changes → `gbr/2`. Agents and apps advertise `proto` and refuse mismatch.
